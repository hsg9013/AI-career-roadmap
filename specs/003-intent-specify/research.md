# Phase 0 Research — 003 실연동·안정화

모든 NEEDS CLARIFICATION은 spec의 Clarifications(2026-06-10)에서 해소됨. 본 문서는 각 연동의 결정·근거·대안을 정리한다.

## R-1. AI 모델 제공자 — Claude 최신 모델

- **Decision**: 추천·문서·미션 피드백 생성을 Anthropic Claude 최신 모델(Opus 4.x = 고난도 분석, Sonnet 4.x = 일반 생성)로 연동. `@anthropic-ai/sdk` 사용, `services/ai/`에 클라이언트·프롬프트·토큰/비용 가드 캡슐화.
- **Rationale**: 프로젝트 환경 기본값이자 한국어·장문 추론 강점. 모델 ID는 환경변수로 분리해 교체 가능하게 둔다.
- **Alternatives**: OpenAI GPT(별도 키·정책 관리), 자체 오픈소스 호스팅(운영 부담↑) — 모두 기각.
- **연동 패턴**: 모든 AI 호출은 `(입력)→Claude→검증·정규화→결과` 또는 실패 시 `(입력)→규칙기반→결과`. 호출은 타임아웃(로드맵 < 2.0s 목표)·재시도 1회·비용 한도 가드를 통과해야 하며, 부분 응답은 누락 항목만 규칙기반으로 보완.

## R-2. 폴백·비용 가드 정책

- **Decision**: 일/월 토큰 예산 상한과 요청당 타임아웃을 환경설정으로 두고, (a)호출 실패 (b)타임아웃 (c)예산 초과 시 규칙 기반 결과로 100% 폴백. 폴백 발생을 `ai_inference_log`와 메트릭으로 관측.
- **Rationale**: 사용자에게 빈 결과·오류 노출 0(SC-006), 비용 통제, 점진 활성화 원칙.
- **Alternatives**: 폴백 없이 오류 노출(UX·신뢰 저하) 기각.

## R-3. 직무·산업 사전 — 산업 10·직무 ~50

- **Decision**: `catalog` 모듈에 산업(10)·직무(산업당 5내외, 총 ~50)·직무별 핵심 역량 키워드를 정규화 저장하고 시드로 적재. 회원가입 선택지·갭 진단 기준의 단일 출처(SSOT).
- **Rationale**: 외부 의존이 없어 즉시 착수(P1), 진단·추천 적용 범위 확대.
- **Alternatives**: 외부 직무 표준(예: 워크넷 코드) 직접 동기화 — 초기 범위 과대, 후순위로 보류.
- **마이그레이션**: 기존 5직무(IT 4·금융 1)를 신규 스키마로 이관 후 확충.

## R-4. 결제·정산 실연동 — PortOne

- **Decision**: PortOne 결제(승인/취소/환불) 실연동 + 웹훅 수신. 결제·멤버십·정산 상태를 멱등 키로 일관 처리하고, 멘토·선배 정산은 기존 정산 데이터 구조 위에서 적립.
- **Rationale**: 002에서 정산 구조는 구현됨 → 결제 흐름만 그 위에 안착(spec 의존성).
- **Alternatives**: 타 PG — 002 결정(PortOne) 계승.
- **안전장치**: 웹훅 서명 검증, 재수신 멱등, 승인·활성화 불일치 자동 정정/보류→운영 검수.

## R-5. 알림 3채널 — 앱/FCM·APNs/Naver Mail

- **Decision**: 이벤트→채널 라우터가 사용자 채널 설정을 읽어 앱 내 알림(항상)·푸시(FCM/APNs)·메일(Naver Cloud)로 발송. BullMQ 잡으로 비동기 처리·재시도.
- **Rationale**: 재방문·리텐션(SC-009), 외부 발송 실패 격리.
- **Alternatives**: 동기 발송(지연·실패 전파) 기각.
- **안전장치**: 앱 내 알림 보장, 실패 채널 재시도 후 누락 기록, 채널 off 반영.

## R-6. 외부 데이터 수집 — 공식 API·제휴 피드만

- **Decision**: 채용·자격증·공모전 데이터를 공식 오픈API·제휴 피드에서만 BullMQ 일일 스케줄로 수집. 항목별 수집 시각·신선도 저장, 기준 기간 초과 시 '최신 아님' 표시. **공개 페이지 크롤링 금지.**
- **Rationale**: ToS·저작권·법적 위험 회피(Clarification), 신선도 보장(FR-011).
- **Alternatives**: 크롤링 병행(법적 위험↑) 기각, 수동 큐레이션(자동화 약함) 보조 수단으로만.
- **안전장치**: 수집 실패 시 기존 데이터 유지·운영자 경보.

## R-7. 소셜 로그인·학교 이메일 검증 — 네이버 OAuth · .ac.kr

- **Decision**: 네이버 OAuth로 계정 생성·연결·로그인. 학교 소속은 `.ac.kr` 이메일로 토큰 메일 검증, 비-`.ac.kr`은 거부. 기존 이메일/비밀번호 로그인은 유지(폴백).
- **Rationale**: 가입 편의·대학 소속 신뢰도.
- **Alternatives**: 다중 소셜(구글·카카오) — 초기엔 네이버만(국내 비중), 확장 여지.

## R-8. 모바일 스토어 배포 — Capacitor 단일 빌드

- **Decision**: 동일 Capacitor 6 빌드를 iOS·Android 스토어에 배포. 푸시 실연동(@capacitor/push-notifications), 배포 자동화(fastlane 등), 모바일 E2E(Detox/Maestro) 도입.
- **Rationale**: "웹+모바일 동시 제공"(SC-012), 코드 1벌 유지.
- **Alternatives**: 네이티브 별도 개발 기각(중복 비용).
- **안전장치**: 스토어 심사 반려가 웹 운영에 무영향.

## R-9. 운영 안정성 — 백업·복구·SPOF·모니터링

- **Decision**: MariaDB 일일 백업(보관 정책)·복구 절차 문서화 및 24h 복구 리허설, 단일 DB SPOF를 standby(복제)로 보완, 헬스·핵심 지표 모니터링과 운영자 경보. 목표 월 99.5%.
- **Rationale**: 결제·AI 실연동이 들어오는 만큼 장애 파급↑ → 전 과정 병행(US8).
- **Alternatives**: 즉시 풀 HA(과투자) 보류 → standby부터 단계적.
- **Observability**: 폴백률·결제 성공률·발송 성공률·수집 성공률·가용성 대시보드.
