## 비즈니스 모델 요약 (참조용)

Intent-Specify.md에 정의된 비즈니스 인텐트를 시스템 명세 작성 전 명료화한다. 본 섹션은 후속 FR·SC·Entity 도출의 근거다.

- **핵심 가치 제안**: 단순 채용 정보 나열이 아닌, 선배 합격 경로 데이터와 현직자 피드백을 결합한 데이터 기반 맞춤형 커리어 설계 및 포트폴리오 자동화.
- **고객 세그먼트**:
  - 학생 (1차 사용자): 진로 방향 설정, 역량 강화, 포트폴리오 구축.
  - 대학교 취업지원센터·학과 (B2B/B2G): 학생 진로 관리, 취업률 제고.
  - 기업 HR·현직자 (B2B + 멘토 풀): 검증된 인재 조기 발굴, 멘토링 수익화.
  - 선배·졸업생 (데이터 제공자 겸 멘토 후보).
- **핵심 활동 4영역**:
  1. 데이터 기반 역량 분석·진단
  2. 데이터 기반 합격 로드맵 설계
  3. 실무 역량 강화 및 하이브리드 피드백
  4. 상시 이력 관리·문서 자동화 및 진척도 점검
- **수익 모델 (4채널 병행)**:
  1. B2C 학생 직접: 프리미엄 멤버십(로드맵 고급 분석·포트폴리오 진단), 단건 결제(자소서 첨삭·이력서 피드백·멘토링·미션 평가).
  2. B2B/B2G: 대학 취업지원센터·학사 시스템 대상 SaaS 라이선스(연 단위), 기업 대상 직무 맞춤 인재 추천·채용 성사 수수료.
  3. 광고/노출: 기업 인턴십·신입 채용 공고의 타겟 노출.
  4. 제휴 수수료: 외부 교육·자격증·어학 콘텐츠 중개.
- **핵심 자원·이네이블러**: AI 분석·추천 엔진, 익명화 처리된 선배 합격 경로 데이터셋, 현직자 멘토 풀, 대학·기업 파트너 네트워크, 클라우드 데이터 인프라.
- **핵심 위험·장벽**: 학생 데이터 입력의 번거로움, 추천 알고리즘 편향(인기 직무 쏠림), 비인기 전공 표본 부족, 민감 정보 보안, 채용 트렌드 정보 노후화.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 학생: 프로필 입력 및 역량 갭 진단 (Priority: P1)

학생이 전공·학년·관심 직무·보유 자격증·수강 과목·프로젝트·동아리·대외활동 경험 등을 통합 입력하면, 시스템은 목표 직무가 요구하는 역량과 현재 보유 역량을 비교하여 부족한 부분을 시각화한 갭 진단 리포트를 제공한다.

**Why this priority**: 진단 결과는 이후 로드맵 추천, 미션 매칭, 포트폴리오 자동화 모든 단계가 의존하는 기초 데이터다. 이 흐름 없이는 후속 가치 제공이 불가능하므로 MVP 최우선 슬라이스.

**Independent Test**: 학생이 프로필·활동 이력 입력 후 목표 직무 1개를 지정하면, 갭 진단 리포트가 "충족 역량 / 부족 역량 / 우선 보완 영역" 3개 영역으로 시각적으로 제공되는지 검증.

**Acceptance Scenarios**:

1. **Given** 학생이 전공·학년·희망 직무·자격증·프로젝트·동아리 이력을 모두 입력한 상태, **When** 갭 진단을 요청, **Then** 시스템은 목표 직무의 요구 역량 대비 충족/부족 항목을 우선순위와 함께 시각적으로 표시한다.
2. **Given** 학생이 필수 프로필 항목(예: 희망 직무)을 비워둔 상태, **When** 갭 진단을 요청, **Then** 시스템은 누락 항목을 안내하고 진단을 보류한다.
3. **Given** 같은 학생이 활동 이력을 추가한 상태, **When** 진단을 다시 요청, **Then** 갭 진단 결과는 갱신된 데이터를 반영하여 재계산된다.
4. **Given** 학생이 갭 진단 결과의 부족 영역 카드를 선택, **When** "추천 활동 보기"를 누름, **Then** 시스템은 해당 영역을 보완할 추천 자격증·활동 목록으로 이동한다.

### User Story 2 - 학생: 데이터 기반 합격 로드맵 추천 (Priority: P2)

학생이 갭 진단 결과를 받은 뒤, 동일/유사 직무에 합격한 선배들의 익명화된 경로 데이터를 기반으로 시기별(학년·학기 단위) 추천 자격증·공모전·대외활동·인턴십·프로젝트 목록을 단계별 로드맵으로 받는다.

**Why this priority**: "선배 경로 기반 맞춤 추천"이 단순 정보 나열 서비스 대비 핵심 차별화 가치. P1 진단을 실행 가능한 행동으로 전환한다.

**Independent Test**: 진단을 마친 학생이 로드맵 화면에 진입하면 본인 학년/잔여 학기 수 기준의 시기별 추천 항목이 우선순위·예상 소요 기간과 함께 표시되는지 검증.

**Acceptance Scenarios**:

1. **Given** 갭 진단이 완료된 학생, **When** 로드맵을 조회, **Then** 시스템은 학년·학기별 추천 활동을 시간 순서로 배열하고 각 항목에 추천 근거(매칭 선배 수, 직무 매칭도)를 표시한다.
2. **Given** 학생이 추천 항목을 "완료" 또는 "건너뛰기"로 표시, **When** 로드맵 재계산, **Then** 다음 추천 항목과 우선순위가 갱신된다.
3. **Given** 비인기 전공·희소 직무를 선택한 학생, **When** 로드맵을 조회, **Then** 시스템은 유사 직무군에서 보완 추천을 제공하고 "데이터 표본이 적음(±N건)"을 명시한다.
4. **Given** 학생이 로드맵 항목 옆 "왜 이 추천?" 버튼을 누름, **When** 상세 보기, **Then** 시스템은 매칭된 선배 사례 수, 매칭 가중치, 비교 가능한 익명 통계를 보여준다.

### User Story 3 - 학생: 활동 자동 정리 및 포트폴리오/이력서 생성 (Priority: P2)

학생이 입력한 활동 기록을 시스템이 자동으로 분류·태깅하여, 자기소개서·이력서·포트폴리오 작성 시 즉시 활용 가능한 구조화된 문서로 출력한다.

**Why this priority**: "활동 입력의 번거로움"이라는 핵심 장벽을 해소하고 학생이 일상적으로 재방문하는 락인 포인트. P1 진단의 입력 데이터를 결과물로 환원.

**Independent Test**: 학생이 활동 5건 이상을 입력한 뒤 "포트폴리오 생성" 또는 "이력서 출력"을 실행하면, 직무 키워드 기준으로 정렬된 활동 요약과 함께 출력 가능한 문서가 생성되는지 검증.

**Acceptance Scenarios**:

1. **Given** 학생이 활동을 키워드 선택 방식으로 입력, **When** 활동을 저장, **Then** 시스템은 활동을 직무 역량·도메인 태그로 자동 분류한다.
2. **Given** 학생이 목표 직무 1개를 지정, **When** 포트폴리오 생성을 요청, **Then** 시스템은 해당 직무 관련도가 높은 활동 순으로 정렬된 포트폴리오 초안을 생성한다.
3. **Given** 학생이 자기소개서 항목별 키워드를 입력, **When** 작성을 요청, **Then** 시스템은 등록된 활동 데이터를 근거로 한 초안 문장을 제시하며, 인용된 활동을 추적 가능한 형태로 표시한다.
4. **Given** 학생이 생성된 문서를 검토, **When** 내보내기를 실행, **Then** 시스템은 PDF 또는 웹 공유 링크로 문서를 출력하며, 공유 링크는 만료 기한·열람 제한을 설정할 수 있다.
5. **Given** 학생이 이력서 템플릿을 변경, **When** 템플릿 선택 후 저장, **Then** 동일 데이터가 새 템플릿으로 재구성되어 표시된다.

### User Story 4 - 학생: 실무 미션 수행 및 하이브리드 피드백 수신 (Priority: P3)

현직자가 등록한 실무 미션을 학생이 수행해 제출하고, 제출물에 대해 AI 1차 분석과 현직자 심층 코멘트를 결합한 통합 피드백을 받는다.

**Why this priority**: 실전 역량 강화의 핵심이지만 현직자 섭외·검수 체계 안정화 후 단계적 확장.

**Independent Test**: 학생이 1개 미션을 수행하고 제출하면, AI 1차 분석 리포트가 자동 생성되고 이후 현직자 코멘트가 추가되어 통합 피드백 화면에 표시되는지 검증.

**Acceptance Scenarios**:

1. **Given** 학생의 희망 직무와 일치하는 미션이 등록된 상태, **When** 미션 목록을 조회, **Then** 시스템은 직무 매칭도·난이도·예상 소요 시간을 함께 표시해 정렬한다.
2. **Given** 학생이 미션 결과물을 제출, **When** 제출 처리, **Then** 시스템은 AI 1차 분석을 즉시 생성하고 현직자 검수 큐에 등록한다.
3. **Given** 현직자가 5영업일 내 코멘트를 작성·승인, **When** 코멘트 게시, **Then** 학생에게 알림이 전송되고 통합 피드백 페이지에서 AI·현직자 의견이 함께 표시된다.
4. **Given** 5영업일 내 현직자 검수 미완료 제출물, **When** SLA 초과 시점 도달, **Then** 시스템은 다른 현직자에게 재할당하거나 AI 보강 피드백으로 대체 처리한다.

### User Story 5 - 학생: 주기적 진척도 점검 및 알림 (Priority: P3)

방학·학기 말 등 정기 시점에 시스템이 활동 이력·로드맵 달성률을 점검하여 진로 방향 재조정 제안과 임박한 자격증 시험·채용 공고·공모전 일정을 알림으로 제공한다.

**Why this priority**: 장기 사용 유지(retention)와 데이터 최신화의 핵심. 학기 단위 사이클을 갖는 사용자 행태에 맞춘 재방문 유도.

**Independent Test**: 시점 트리거(예: 학기 종료일)가 도달하면 등록 학생에게 진척도 요약 리포트와 다가오는 일정 알림이 자동 발송되는지 검증.

**Acceptance Scenarios**:

1. **Given** 학기 종료 시점, **When** 진척도 점검 트리거, **Then** 시스템은 학생별 로드맵 달성률, 신규 추천, 진로 재조정 옵션을 요약 리포트로 전달한다.
2. **Given** 학생이 관심 자격증·공모전을 등록, **When** 마감일이 7일 이내로 다가옴, **Then** 시스템은 알림을 발송한다.
3. **Given** 학생이 알림 채널(앱 내, 이메일)·빈도를 설정, **When** 설정 저장, **Then** 시스템은 해당 설정에 따라 알림을 발송한다.

---

### User Story 6 - 현직자(멘토): 미션 등록 및 코멘트 작성 (Priority: P3)

현직자는 본인 직무와 경험에 기반한 실무 미션을 등록하고, 배정된 학생 제출물에 코멘트를 작성하며, 보상을 수령한다.

**Why this priority**: 미션·피드백 흐름(US4)의 공급 측. 멘토 풀 확보가 P3 가치 제공의 전제 조건.

**Independent Test**: 현직자 계정으로 미션을 등록하고, 학생 제출이 발생하면 검수 큐에 표시되며, 코멘트 게시 후 보상 적립 내역이 본인 대시보드에 기록되는지 검증.

**Acceptance Scenarios**:

1. **Given** 현직자가 인증된 계정으로 로그인, **When** 미션 등록 폼에 직무·난이도·설명·평가 루브릭을 입력, **Then** 시스템은 운영진 검수 후 미션을 활성화한다.
2. **Given** 학생이 본인 미션에 제출, **When** 검수 큐에 진입, **Then** 시스템은 검수 마감 시각(예: 5영업일 후)을 표시한다.
3. **Given** 현직자가 코멘트 작성·승인, **When** 게시, **Then** 시스템은 보상 적립 내역과 누적 검수 건수를 대시보드에 기록한다.
4. **Given** 현직자가 본인 가용 시간을 설정, **When** 큐 할당, **Then** 시스템은 설정된 시간대 외에는 신규 검수를 자동 배정하지 않는다.

---

### User Story 7 - 대학 관리자: 학생 진로 준비 현황 모니터링 (Priority: P3)

대학교 취업지원센터·학과 관리자는 동의를 받은 소속 학생의 진로 준비 상태(로드맵 달성률, 활동 이력 요약)를 집계 또는 개별 단위로 조회하여 상담·프로그램 운영에 활용한다.

**Why this priority**: B2B/B2G SaaS 수익 채널의 핵심. 대학 파트너십이 학생 유입 채널이자 라이선스 매출원.

**Independent Test**: 대학 관리자 계정으로 동의받은 학생 집단의 진로 준비 통계 대시보드에 접근할 수 있고, 학과·학년·직무군 필터로 분류·내보내기 가능한지 검증.

**Acceptance Scenarios**:

1. **Given** 대학 관리자가 인증된 상태, **When** 대시보드 접근, **Then** 시스템은 동의 범위 내 학생 모수의 직무군별 분포, 평균 로드맵 달성률, 자격증 보유 분포를 보여준다.
2. **Given** 관리자가 학과·학년 필터 적용, **When** 조회, **Then** 시스템은 필터 조건의 통계를 갱신해 표시한다.
3. **Given** 학생이 대학에 대한 데이터 공유 동의를 철회, **When** 다음 조회, **Then** 해당 학생 데이터는 결과에서 제외된다.
4. **Given** 관리자가 통계를 내보내기, **When** 내보내기 요청, **Then** 시스템은 개인 식별 정보가 제거된 형태의 파일을 생성한다.

### User Story 8 - 기업 HR: 직무 요구사항 등록 및 인재 매칭 (Priority: P4)

기업 HR 담당자는 직무 요구사항을 등록하고, 조건에 부합하면서 매칭 동의를 한 학생에게 채용 정보·인턴십 공고를 노출하거나 개별 컨택 요청을 보낸다.

**Why this priority**: B2B 채용 연계 수수료와 광고 수익 채널. P3 흐름의 데이터·로드맵이 일정 수준 누적된 뒤에 가치 발생.

**Independent Test**: 기업 HR 계정으로 직무를 등록하고, 매칭 조건을 만족하는 동의 학생 풀의 익명 통계를 확인할 수 있으며, 컨택 요청 발송 시 학생이 수락 시 식별 정보가 공개되는지 검증.

**Acceptance Scenarios**:

1. **Given** 기업 HR이 인증된 상태, **When** 직무·요구 역량·근무 조건을 등록, **Then** 시스템은 매칭 모수의 익명 집계(N명, 평균 매칭 점수)를 보여준다.
2. **Given** HR이 채용 공고를 게시, **When** 조건 일치하는 동의 학생 풀에 노출, **Then** 학생에게 알림이 발송된다.
3. **Given** HR이 특정 익명 학생에게 컨택 요청, **When** 학생이 수락, **Then** 시스템은 학생의 동의된 식별 정보(이름·이메일·이력서)를 HR에게 공개한다.
4. **Given** 채용 성사 신고가 등록, **When** 처리 완료, **Then** 시스템은 합의된 채용 수수료 청구 내역을 생성한다.

### User Story 9 - 학생: 멤버십 결제 및 단건 결제 (Priority: P4)

학생은 무료 기능을 사용해보다가 프리미엄 멤버십을 구독하거나, AI 자소서 첨삭·현직자 멘토링·미션 평가 등 단건 서비스를 결제한다.

**Why this priority**: B2C 직접 수익의 게이트. 무료 사용자가 가치를 인지한 뒤 결제로 전환되는 흐름이므로 P1·P2·P3 안정화 후 확장.

**Independent Test**: 무료 학생이 프리미엄 기능을 시도하면 결제 안내가 표시되고, 결제 완료 후 즉시 프리미엄 기능이 활성화되며 결제 이력에서 영수증을 받을 수 있는지 검증.

**Acceptance Scenarios**:

1. **Given** 무료 사용자가 프리미엄 전용 기능(예: 고급 로드맵 분석)에 접근 시도, **When** 결제 안내 표시, **Then** 시스템은 멤버십·단건 옵션과 가격을 보여준다.
2. **Given** 사용자가 결제 완료, **When** 결제 승인, **Then** 시스템은 해당 권한을 즉시 활성화하고 영수증을 발송한다.
3. **Given** 사용자가 멤버십을 해지, **When** 해지 요청, **Then** 시스템은 결제 주기 종료 시점까지 권한을 유지하고 자동 갱신을 중단한다.
4. **Given** 결제 실패(카드 거절 등), **When** 실패 처리, **Then** 시스템은 사용자에게 사유와 재시도 방법을 안내한다.

### User Story 10 - 선배(졸업생/취업자): 합격 경로 데이터 기부 및 익명화 동의 (Priority: P4)

취업에 성공한 선배가 본인의 이력서·포트폴리오·활동 경로를 익명화 동의 하에 플랫폼에 기부하여 추천 엔진의 학습 데이터에 기여하고, 기여 대가로 인센티브(현금·뱃지·서비스 크레딧)를 받는다.

**Why this priority**: 추천 알고리즘의 품질을 결정하는 데이터 공급원. 초기 콜드 스타트 해소에 결정적.

**Independent Test**: 선배 계정으로 합격 경로 데이터 업로드 및 익명화 동의 후, 본인 데이터가 추천 통계에 반영되었음을 확인하고, 인센티브 적립 내역을 볼 수 있는지 검증.

**Acceptance Scenarios**:

1. **Given** 선배가 인증·동의 완료, **When** 이력서·합격 정보 업로드, **Then** 시스템은 식별 정보를 분리·익명화 처리하고 데이터셋에 추가한다.
2. **Given** 선배가 동의를 철회, **When** 처리, **Then** 시스템은 해당 사용자의 식별 가능한 원본 데이터를 삭제하되, 이미 학습에 반영된 통계는 유지하고 그 사실을 명시한다.
3. **Given** 선배 데이터가 N건 이상 활용, **When** 인센티브 적립 조건 충족, **Then** 시스템은 보상을 적립하고 내역을 공개한다.

---

### Edge Cases

- **비인기 전공·희소 직무 표본 부족**: k-익명성 임계치(k≥5)를 충족하지 못하는 직무군 추천 시 시스템은 "데이터 표본 부족"을 명시하고 유사 직무군 기반 보완 추천을 제공한다.
- **민감 정보 누출 시도**: 학적·성적·상세 활동 이력에 권한 없는 접근 시도 시 시스템은 차단 및 감사 로그 기록.
- **AI 추천 거부 누적**: 학생이 추천을 N회 이상 연속 거부할 때 시스템은 추천 가중치를 재조정하고 대안 직무군을 제시한다.
- **현직자 코멘트 미수신**: 미션 제출 후 5영업일 내 검수 미완료 시 재할당 또는 AI 보강 피드백으로 대체 처리.
- **활동 데이터 중복·누락**: 동일 활동의 중복 입력 또는 핵심 정보 누락 시 자동 보정 또는 재입력 요청.
- **시장 정보 노후화**: 채용 공고·자격증 일정 등 외부 데이터가 24시간 이상 갱신되지 않을 때 "최신 정보 아님" 표시.
- **포트폴리오 비교 부담**: 사용자 간 직접 비교 기능은 기본 비활성, 노출 범위는 사용자가 제어.
- **선배 데이터 익명화 무결성**: 활동 경로 조합만으로 개인 식별 위험이 있을 때 k-익명성 미달 데이터는 추천 풀에서 제외.
- **결제 실패·이중 청구**: 결제 게이트웨이 오류 발생 시 사용자에게 안내하고, 중복 청구는 운영자 검토 후 자동·수동 환불.
- **학생 동의 철회 시 파트너 데이터 영향**: 학생이 대학·기업과의 데이터 공유 동의를 철회하면 다음 조회 시점부터 즉시 해당 데이터셋에서 제외, 단 이미 학습된 익명 통계는 유지.
- **법적 채용 차별 우려**: 매칭 알고리즘이 성별·연령·출신 학교 등 민감 속성으로 차별하지 않도록 해당 속성을 매칭 입력에서 배제한다.
- **외부 데이터 소스 장애**: 외부 채용·자격증 데이터 소스 장애 시 마지막 정상 데이터 + 장애 표시로 운영 지속.

## Requirements *(mandatory)*

### Functional Requirements

#### 1. 사용자 계정 및 권한

- **FR-001**: 시스템은 학생, 현직자(멘토), 기업 HR, 대학 관리자, 운영자 5종 역할을 구분하여 권한을 부여해야 한다.
- **FR-002**: 시스템은 이메일·비밀번호 기반 인증과 소셜 로그인(Google, 네이버)을 지원하며, 학생 계정은 학교 이메일(.ac.kr) 검증으로 학적 신뢰성을 확보한다.
- **FR-003**: 시스템은 현직자·기업 HR 계정에 대해 재직 증명(명함, 재직증명서 업로드 등)을 거쳐 운영자가 검수·승인한 뒤 활성화한다.
- **FR-004**: 학생은 본인 데이터에 대해 다음 4가지 권한을 행사할 수 있어야 한다 — (a) **열람**: 마이페이지에서 본인 보유 데이터 전체 조회, (b) **수정**: 프로필·활동·동의 항목 수정, (c) **내보내기**: 본인 모든 도메인 데이터(프로필·활동·로드맵·문서·미션·결제·동의 이력)를 1회 요청으로 ZIP+JSON 패키지로 다운로드(7일 유효 보관), (d) **삭제(탈퇴)**: 탈퇴 요청 시 즉시 비활성화 + 30일 유예 기간 + 유예 종료 시 식별 컬럼 익명화·관계 데이터 보존(k-익명성 충족 분만 추천 풀 잔존). 시스템은 졸업 후 1년까지 활성 사용자 데이터를 보관한 뒤 명시적 연장 요청이 없으면 야간 잡으로 자동 익명화·삭제하며 사전 90일·30일·7일에 알림한다.

#### 2. 프로필 및 역량 진단

- **FR-005**: 시스템은 학생이 전공, 학년, 희망 산업·직무, 보유 자격증, 수강 과목, 프로젝트·동아리·대외활동·인턴십·수상 이력을 등록할 수 있도록 해야 한다.
- **FR-006**: 시스템은 학생의 입력 부담을 줄이기 위해 키워드 선택 방식의 간편 입력 인터페이스를 제공하며, 자유 텍스트 입력도 병행 지원한다.
- **FR-007**: 시스템은 학생의 현재 역량과 선택한 목표 직무의 요구 역량을 비교하여 갭 분석 결과를 시각적(레이더 차트·항목 리스트 등)으로 제시해야 한다.
- **FR-008**: 학생은 진단 결과의 각 부족 영역에서 "추천 활동 보기"로 즉시 후속 추천 흐름으로 이동할 수 있어야 한다.
- **FR-009**: 학생은 목표 직무를 최대 3개까지 등록할 수 있으며, 시스템은 각 목표별 진단·로드맵을 별도로 제공한다.

#### 3. 합격 로드맵 추천

- **FR-010**: 시스템은 익명화된 선배 합격 사례(학기별 스펙, 활동 순서, 포트폴리오 구성)를 분석하여 학생별 맞춤 로드맵을 생성해야 한다.
- **FR-011**: 로드맵은 학년·학기 시계열에 추천 자격증·공모전·대외활동·인턴십·프로젝트를 배치하고, 각 항목의 추천 근거(매칭 선배 수, 직무 적합도, 표본 신뢰도)를 표시해야 한다.
- **FR-012**: 학생은 추천 항목을 완료·진행 중·건너뛰기로 표시할 수 있어야 하며, 시스템은 표시에 따라 로드맵을 재계산해야 한다.
- **FR-013**: 시스템은 비인기 전공·희소 직무의 경우 표본 부족(k<5)을 명시하고 유사 직무군 기반 보완 추천을 제공해야 한다.
- **FR-014**: 시스템은 학생이 추천을 거부한 이력을 학습하여 후속 추천에서 유사 항목의 가중치를 낮춰야 한다.
- **FR-015**: 시스템은 추천 결과가 성별·연령·출신 학교 등 민감 속성에 의해 편향되지 않도록 해당 속성을 매칭 입력에서 배제해야 한다.

#### 4. 활동 기록 자동 분류 및 문서 생성

- **FR-016**: 시스템은 학생이 입력한 활동을 직무 역량·도메인 키워드로 자동 분류·태깅해야 하며, 학생이 태그를 수정·추가할 수 있어야 한다.
- **FR-017**: 시스템은 학생의 활동 데이터를 기반으로 이력서·자기소개서·포트폴리오 초안을 직무 적합도 순서로 자동 생성해야 한다.
- **FR-018**: 자기소개서 초안은 인용한 활동 ID를 명시적으로 표시하여 학생이 출처를 검증·편집할 수 있어야 한다.
- **FR-019**: 학생은 자동 생성된 문서를 편집할 수 있어야 하며, 시스템은 PDF 다운로드와 만료 기한·열람 제한 가능한 웹 공유 링크 형태의 내보내기를 지원해야 한다.
- **FR-020**: 시스템은 복수 이력서·포트폴리오 템플릿을 제공하며, 학생이 템플릿을 전환해도 데이터는 동일하게 유지되어야 한다.

#### 5. 실무 미션 및 하이브리드 피드백

- **FR-021**: 현직자는 본인 직무에 기반한 실무 미션(직무, 난이도, 설명, 평가 루브릭)을 등록할 수 있어야 하며, 운영자 검수를 거쳐 활성화된다.
- **FR-022**: 학생은 미션 결과물을 제출할 수 있고, 시스템은 제출 즉시 평가 루브릭 기반 AI 1차 분석 리포트를 생성해야 한다.
- **FR-023**: 현직자가 작성한 코멘트는 AI 분석과 통합된 단일 피드백 페이지에 표시되어야 한다.
- **FR-024**: 시스템은 현직자 검수가 5영업일 내 완료되지 않으면 자동으로 다른 현직자에게 재할당하거나 AI 보강 피드백으로 대체 처리해야 한다.
- **FR-025**: 시스템은 현직자별 누적 검수 건수·평균 검수 시간·학생 만족도(평점)를 기록하고, 일정 임계치 미달 시 신규 배정을 제한해야 한다.

#### 6. 진척도 점검 및 알림

#### 9. 기업 파트너 인터페이스 (B2B 채용·광고)

- **FR-037**: 기업 HR은 직무 요구사항(역량·근무 조건·채용 기간)을 등록하고, 매칭 모수의 익명 집계(N명, 평균 매칭 점수)를 즉시 확인할 수 있어야 한다.

- **FR-041**: 시스템은 광고 형태로 노출되는 채용·인턴십 공고를 일반 추천 항목과 시각적으로 구분되게 표시해야 한다(예: "Sponsored" 배지).

#### 10. 결제·멤버십·인센티브 (B2C 수익 + 멘토 보상)

## Success Criteria *(mandatory)*

### Measurable Outcomes

#### 학생 핵심 흐름
- **SC-001**: 학생이 프로필·활동 이력 최초 입력부터 첫 갭 진단 리포트 수신까지 평균 15분 이내에 도달할 수 있다.
- **SC-002**: 신규 가입 학생이 첫 7일 내 프로필 입력 완료, 갭 진단 1회, 로드맵 조회 1회의 세 가지 핵심 행동을 모두 수행하는 비율이 50% 이상.
- **SC-003**: 학기 단위 활성 학생 중 70% 이상이 로드맵 추천 항목을 최소 1건 "완료"로 표시.
- **SC-004**: AI 추천에 대한 학생의 명시적 거부율(부적합 표시)이 30% 이하로 유지.
- **SC-005**: 학생의 포트폴리오 자동 생성 결과를 90% 이상의 학생이 다운로드 또는 공유 링크로 1회 이상 외부 활용.
- **SC-006**: 미션 제출 후 통합 피드백이 5영업일 내 학생에게 전달되는 비율이 80% 이상.
- **SC-007**: 비인기 전공·희소 직무 학생군에 대해서도 추천 만족도가 평균 추천 만족도 대비 ±10%p 이내로 유지.

#### 데이터 품질·보안
- **SC-008**: 추천 풀에 사용되는 선배 합격 데이터셋의 k-익명성이 100% k≥5 충족.
- **SC-009**: 민감 데이터의 권한 외 접근 시도는 100% 차단·로깅되며, 외부 유출 사고는 0건 유지.
- **SC-010**: 채용 공고·자격증 일정 등 외부 데이터의 평균 갱신 지연이 24시간 이내로 유지.

#### 파트너·B2B
- **SC-011**: 대학 라이선스 계약 후 평균 10영업일 이내에 학생 계정·동의 흐름 연동이 완료.
- **SC-012**: 대학 관리자 월간 활성 사용률(MAU 기준 로그인 이상)이 계약 시트의 60% 이상.
- **SC-013**: 기업 HR이 직무 등록 후 매칭 학생 풀 통계 확인까지 평균 3분 이내.
- **SC-014**: 기업 컨택 요청 → 학생 수락 → 매칭 성사 전환율(funnel)을 월 단위로 측정·공개할 수 있다.

#### 수익·운영
- **SC-015**: 무료 학생 사용자 중 가입 후 90일 내 멤버십 또는 단건 결제 1회 이상 발생 비율이 8% 이상.
- **SC-016**: 멤버십 월간 해지율(churn)이 5% 이하 유지.
- **SC-017**: 멘토 풀의 월간 활성 멘토 수가 활성 학생 100명당 최소 1명 비율 유지.
- **SC-018**: 학기 종료 시점 자동 알림을 받은 학생의 30일 내 재방문율이 60% 이상.
- **SC-019**: 결제 실패·환불 처리에서 사용자가 7일 내 자동 안내를 받는 비율이 100%.

#### 추천 품질
- **SC-020**: 학생이 "완료"로 표시한 추천 항목의 직무 매칭 점수가 거부 표시 항목보다 평균 20%p 이상 높음(추천 알고리즘의 변별력 지표).

## Assumptions

### 비즈니스 모델·범위
- **1차 출시 플랫폼**: 반응형 웹과 모바일 앱(iOS/Android)을 **동시에 1차 출시 범위에 포함**한다. 백엔드 API는 단일 코드베이스, 프런트엔드는 `frontend-web/`(Vite+Vue 3.4) 및 `frontend-mobile/`(크로스플랫폼 프레임워크 — plan.md 결정 사항) 두 워크스페이스로 분리하며, 공통 API 타입은 `shared/types/`로 공유한다.
- **다국어**: 초기 출시는 한국어만 지원. 영문(외국인 유학생)·기타 언어는 후속.
- **법·규제**: 한국 개인정보보호법 및 정보통신망법을 1차 준수 대상으로 하며, 해외 사용자의 데이터에도 동일 정책을 적용한다.

### 데이터 출처·운영
- **학생 데이터**: 본인 입력 데이터를 1차 소스로 하며, 대학 학사 시스템 연동은 파트너 계약 후 단계적으로 도입.
- **선배 데이터 수집**: 동의를 받은 졸업생·취업 성공자로부터 익명화 처리 후 수집하고, k-익명성(k≥5)을 충족해야만 추천 풀에 포함.
- **외부 채용·자격증 데이터**: 공식 API·파트너 피드·웹 스크래핑(이용약관 허용 범위 내)을 통해 합법적으로 일 단위 이상 갱신.
- **현직자 멘토 수급**: 멘토 보상은 **하이브리드 모델**을 채택한다 — (a) **정액 트랙**: 장기 계약 멘토에게 세션당 고정 단가(예: 30분 30,000원, 단가는 운영 정책으로 조정 가능) 지급, (b) **수수료 트랙**: 학생이 일회성 세션 비용을 결제하면 플랫폼이 수수료(잠정 20%, 운영 정책으로 조정 가능)를 공제한 잔액을 멘토에게 지급. 두 트랙 모두 전문성 뱃지·평점 시스템을 공유하며, 멘토는 가입 시 트랙을 선택하고 계약 갱신 주기마다 변경 가능. 정산 도메인(`mentor_payouts`, `commission_rates`, `payment_methods`)·PG 연동·세금 처리는 plan.md/data-model.md에서 상세 설계.

### 사용자 환경·표준
- 학생은 안정적 인터넷 환경에서 최신 브라우저(Chrome, Safari, Edge, Firefox 최신 2개 메이저 버전)로 서비스를 이용.
- 인증은 이메일·비밀번호 + 소셜(Google·네이버) 기본 제공. 학교 SSO는 대학 파트너 계약 시 단계적 도입.
- 알림은 앱 내 알림과 이메일 기본 제공. SMS·카카오 알림톡은 멤버십 부가 기능 또는 향후 확장 옵션.

### 결제·수익 운영
- 결제는 외부 결제 게이트웨이를 통해 수행되며, 영수증·환불·VAT 처리 등 표준 절차를 따른다.
- 채용 매칭 수수료·광고 단가·제휴 수수료는 파트너 계약별로 협의되며, 시스템은 약정에 따른 정산 자료만 생성한다.
- 가격·요금제는 본 명세 범위 외이며, 별도 가격 정책 문서에 따른다.

### 거버넌스·문서 범위
- 본 명세는 기술 스택·API 설계·코드 구조를 규정하지 않으며, 비즈니스 요구만 다룬다. 기술 결정은 `plan.md`에서 다룬다.
- 단계적 출시: P1(US1) → P2(US2·US3) → P3(US4·US5·US6·US7) → P4(US8·US9·US10) 순. 각 단계는 이전 단계 안정화 후 진입.
- 대학·기업 파트너 인터페이스(US7·US8)는 학생 핵심 흐름(P1·P2)이 자체 사용자 베이스를 확보한 뒤 가치 발생.
